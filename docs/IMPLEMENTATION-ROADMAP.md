# IMPLEMENTATION ROADMAP · Learning Companion

## 1. 文档目标

这份路线图描述的是：

**如何基于当前已经完成的桌面应用基线，把产品推进到“人格画像驱动的 AI 学习规划助手”。**

它不再沿用上一轮“Learning Cockpit 已收口、下一步 Release Candidate”的判断，而是把当前代码基线视为新的出发点，进入下一阶段的产品改造。

- **当前阶段：Phase 6 / 桌面角色反馈层（Task 4 已完成）**
- **当前唯一下一任务：Release Track / Task 1 / 回归、打包和发布准备**

- 今日步骤现已具备即时流转反馈、下一步自动上浮高亮，以及延期进入“明天候选区”的目标区域脉冲提示。
- 日历页现会监听 `weeklyPlan / delayedPlacements` 的变化，展示“日历已重排”反馈卡，并对受影响的主目标连续块、副目标补位块和延期补回卡追加流动式过渡与迁移原因说明。
- renderer store 现已提供短时 `companionCue` 通道，今日页的生成/步骤反馈与日历页的重排反馈都会先推送给桌面角色，再回落到基于 `dashboard` 的常规陪伴层总结。

### 当前最关键的差距
- 桌面角色已经具备跨页面联动，但后续动作、人设语气和扩展接口还没有被结构化收口。

## 3. 执行总策略

1. 先把“理解用户”和“替用户做选择”这两个核心价值做实，再谈更复杂的形象和玩法。
2. 先改造今日页为执行页，再补花哨视觉层；因为日常价值必须先成立。
3. 多目标必须以“一个主目标 + 多个副目标”为核心，不做完全平权并行。
4. 领域扩展先靠通用规划内核 + 少数领域深做，不追求一开始覆盖所有学习类型。
5. 桌面动漫形象是强化体验层，不是当前主流程的阻塞项。

## 4. 执行规则

### Rule 1: 同一时刻只允许一个主阶段处于 `in_progress`
- 可以有小型并行修复。
- 不允许同时推动两个主产品阶段。

### Rule 2: 每个阶段必须带验收条件
- 验收必须包括用户路径、状态持久化、重启回填和错误反馈。
- 没有验收条件的阶段不能进入 `in_progress`。

### Rule 3: 每个阶段结束后必须指定唯一下一任务
- 阶段文档和进度记录必须显式写出 `Next Task`。
- 没有唯一下一任务，阶段不算完成。

### Rule 4: 阶段性交付必须进入 git
- 每个阶段至少有一次可回溯提交。
- commit message 应体现业务价值，而不是泛泛描述。

### Rule 5: 先做结构化边界，再做体验加强
- 状态模型、页面职责和调度规则先清楚。
- 动效、桌面角色和表现层增强放在后续阶段叠加。
