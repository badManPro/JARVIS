# Personality-Driven Learning Companion Design

## 1. 背景与目标

当前项目已经具备桌面端学习规划产品的基本骨架，包括：
- 首次启动 onboarding
- 学习档案页面
- 粗版学习路径页面
- 今日细版计划页面
- 本地持久化与 AI capability 抽象

本轮产品设计的目标不是新增一个独立模块，而是把现有项目升级为一个更明确的产品：

**一个基于人格画像、时间预算和真实执行反馈的 AI 学习规划助手。**

它不是课程平台，不是纯聊天助手，也不是普通待办工具。它的核心价值是：
- 先理解用户，再做计划
- 在用户真正要开始学习时，生成极细的执行方案
- 尽可能减少用户在“今天该做什么、先做什么、去哪里学”的决策成本

用户优先价值排序已经确认：
1. 计划足够细，尽量替用户做掉选择
2. AI 要“懂用户”，并根据性格、MBTI、反馈偏好和状态变化做调整

## 2. 产品定位

产品首发定位为：

**通用学习规划内核 + 少数重点领域深做**

首发支持的重点领域：
- 编程
- 乐器
- 健身

原因：
- 三者都适合拆成“粗路径 + 当天执行单”
- 三者都适合依据画像和时间预算动态调整
- 三者都能明确产出任务原子、资源推荐和执行反馈

产品不是只面向编程学习者，后续可以通过扩展“领域规则 + 资源规则 + prompt few-shot”支持更多领域，例如绘画、英语、摄影或写作。

## 3. 核心产品原则

### 3.1 先建画像，再给计划
首次使用时，系统需要建立一个轻量但真实有用的画像，而不是简单收集资料。

采集字段包括：
- 年龄阶段
- MBTI
- 时间预算
- 最佳学习窗口
- 节奏偏好
- 动机方式
- 压力反应
- 反馈偏好
- 个性关键词

系统不会只做表单收集，还需要补一层“规划倾向确认”，告诉用户：
- 系统会如何拆任务
- 系统会如何提醒
- 系统何时自动替用户做决定

### 3.2 计划分层，避免长期和当天混在一起
产品中的计划必须分成两层：

1. 粗版路径
   - 长周期主线
   - 阶段目标
   - 周里程碑
   - 关键检查点

2. 细版执行单
   - 只在当天或临近执行时生成
   - 包含顺序、时长、资源、练习、完成标准和今日产出

粗版负责方向，细版负责落地。

### 3.3 多目标存在，但只有一个主目标
用户可以管理多个目标，但默认采用：

**一个主目标 + 多个副目标**

调度原则：
- 主目标优先保证连续推进
- 副目标占用剩余时间和精力
- 当可用时间变少时，优先保主目标
- 系统尽量减少目标切换成本

### 3.4 小调整自动执行，大调整再确认
AI 具备有限自治权：

自动执行的小调整：
- 今天时间从 60 分钟变成 30 分钟
- 用户状态差，需要自动降难度
- 某一步延期，需要顺延到明天候选区

需要确认的大调整：
- 切换主目标
- 重新分配一周主副目标权重
- 推翻现有粗版路径

## 4. 页面结构设计

当前一级导航结构可继续保留，并补充一个日历排程页：
- 今日
- 学习路径
- 学习档案
- 日历
- 设置

### 4.1 今日页
今日页是产品的执行核心，不应只是计划展示页。

今日页职责：
- 输入今天可用时间、窗口和限制
- 生成当天细版执行单
- 管理执行步骤状态流转
- 展示今日资源、练习和交付物
- 提供即时调整入口

细版计划生成后，每个步骤都应支持状态流转：
- 待开始
- 进行中
- 完成
- 延期
- 跳过

其中已确认的重要规则：
- `延期` 默认自动顺延到明天候选区
- 完成一步后，系统自动把下一步提升为主焦点
- 跳过一步时，如果影响后续依赖，需要提醒并压缩或重排后续步骤

交互目标：
- 用户完成一步后立即得到反馈
- 用户不需要自己判断“下一步该干什么”
- 页面能提供明显的进度推进感

### 4.2 学习路径页
学习路径页只承担粗版计划管理：
- 阶段目标
- 周里程碑
- 关键节点
- 主目标与副目标关系
- 粗版重生成

这里不承接过细的今日任务，避免长期路径被日执行细节淹没。

### 4.3 学习档案页
学习档案页不仅是编辑资料的地方，还应明确说明“系统如何使用这些信息”。

页面应展示：
- 年龄阶段、MBTI、时间预算、窗口等画像字段
- 规划风格解释
- 反馈方式解释
- 系统自动调度偏好

建议新增：
- “当前规划倾向”区块
- “这些设定会如何影响计划”区块

### 4.4 日历页
新增日历页，用于承接多目标调度结果。

日历页展示：
- 一周时间块
- 主目标优先占位
- 副目标的补位安排
- 延期任务进入明天候选区后的再安排
- 冲突时系统如何挪动任务

这个页面的价值不在于手动管理日程，而在于让用户看见系统是如何替自己安排时间的。

### 4.5 设置页
设置页继续承接 Provider 与模型策略，但应增加产品级设置：
- 自动小调整开关
- 今日计划默认细化程度
- 资源偏好
- 提醒风格

## 5. AI 规划引擎设计

建议把 AI 规划系统拆成两层：

### 5.1 通用规划内核
负责处理与领域无关的能力：
- 读取画像和规划倾向
- 判断主副目标优先级
- 根据时间预算做日级调度
- 控制当天计划强度
- 根据执行结果和复盘更新后续安排

通用规划内核的输入包括：
- 用户画像
- MBTI 与行为倾向
- 主目标 / 副目标
- 时间预算
- 最佳学习窗口
- 最近完成率
- 压力和节奏变化
- 历史延期 / 跳过记录

### 5.2 领域执行器
每个领域需要有独立的执行规则，首发深做三类：

#### 编程
- 学哪些语法点
- 做几个例子
- 看哪个官方文档或教程
- 做什么最小练习
- 今日交付什么代码结果

#### 乐器
- 先热身什么
- 练哪段
- 节拍器速度是多少
- 每段重复几次
- 是否录音回听

#### 健身
- 训练主题
- 动作顺序
- 组数 / 次数 / 强度
- 休息时长
- 恢复提醒和次日影响

后续扩展更多领域时，不应该推翻通用结构，而应增加新的“领域执行器”配置：
- prompt 规则
- few-shot 示例
- 推荐资源来源
- 任务原子模板

## 6. 数据模型扩展建议

当前 `src/shared/app-state.ts` 已有较好的基础，但需要扩展以下方向：

### 6.1 UserProfile
在现有字段基础上，建议新增：
- `planningStyle`
- `decisionSupportLevel`
- `feedbackTone`
- `autonomyPreference`

目的：
- 不只保存 MBTI 本身
- 保存系统确认后的“可执行规划倾向”

### 6.2 LearningGoal
建议新增：
- `domain`
- `role`（main / secondary）
- `scheduleWeight`

目的：
- 支持通用领域扩展
- 明确主副目标关系
- 为时间调度提供权重基础

### 6.3 TodayPlan
当前 `TodayPlan` 需要从“静态展示结构”升级成“执行会话结构”。

建议细化：
- step 级状态
- step 级开始时间 / 完成时间
- 延期原因
- 是否自动顺延
- 与后续步骤的依赖关系
- 系统是否已重排后续任务

### 6.4 Calendar Scheduling State
建议增加一套排程实体，用于保存：
- 某天主目标占位
- 某天副目标补位
- 延期候选任务
- 自动调度结果
- 用户确认过的排程变更

## 7. 核心交互反馈设计

产品体验重点不是“做出更多表单”，而是做出更强的反馈感。

建议重点投入的反馈包括：
- 生成今日计划时的 staged animation
- 完成步骤后的即时反馈
- 自动把下一步顶上来的焦点切换
- 延期后任务流向明天候选区的动画提示
- 重排多目标时间块时的流动式过渡

桌面动漫形象可以作为后续强化方向，但不建议在 MVP 阶段作为主入口。

推荐阶段定位：
- Phase 1-4：产品主价值先跑通
- Phase 5：桌面角色作为反馈层和陪伴层加入

首版角色职责：
- 提醒
- 庆祝
- 节奏反馈
- 情绪陪伴

后续再扩展动作和人格表现。

## 8. MVP 实施顺序

### Phase 1：画像与首次建档升级
- 扩展 onboarding 字段
- 新增规划倾向确认
- 把画像字段和系统使用方式解释清楚

### Phase 2：今日页升级为执行页
- 今日细版计划改成步骤级任务结构
- 支持开始 / 完成 / 延期 / 跳过
- 延期默认自动顺延到明天候选区
- 完成后自动推进下一步

### Phase 3：主副目标与日历调度
- 支持一个主目标和多个副目标
- 增加日历页
- 提供基础自动排程

### Phase 4：三类重点领域深做
- 编程规则
- 乐器规则
- 健身规则
- 每个领域补资源来源和 few-shot 样例

### Phase 5：桌面动漫形象
- 作为反馈层加入
- 补动作与提示语
- 不影响主流程

## 9. 成功标准

本轮改造完成后，产品至少需要满足：
- 用户首次启动后，能明确感知“系统已经理解了我”
- 用户不需要自己设计今天学什么，系统能给出足够细的执行单
- 今日步骤可以逐步更新状态，不只是只读展示
- 延期和跳过会进入后续调度逻辑，而不是丢失
- 多目标不会导致日常执行混乱，主目标始终清晰
- 编程、乐器、健身三类领域的计划细节明显不同，不会像通用模板

## 10. 当前结论

本轮已经确认的关键决策如下：
- 产品优先价值是“超细计划”与“理解用户”
- 产品面向通用学习场景，但首发只深做少数重点领域
- 首发深做领域：编程、乐器、健身
- 多目标模式：一个主目标 + 多个副目标
- AI 权限策略：小调整自动执行，大调整再确认
- 今日页必须支持步骤级状态流转
- 延期任务默认自动顺延到明天候选区
- 桌面动漫形象适合放在后续阶段，先作为反馈层而非主交互

## 11. 推荐下一步

建议下一步直接进入实现准备，顺序如下：
1. 先更新共享状态与 onboarding 结构
2. 再重构今日页的 step 执行模型
3. 然后补主副目标与日历调度
4. 最后再做领域深度和角色反馈层
