# PRODUCT PRINCIPLES · Learning Companion

## 1. C 端客户端优先
Learning Companion 的第一落点是 **面向个人用户的桌面客户端**，而不是管理后台、团队系统或复杂工作台。

这意味着界面设计始终遵守以下原则：

### 1.1 一屏先回答“我现在该做什么”
- 首屏优先展示今日任务、当前阶段、风险提醒、建议动作。
- 不把信息架构做成后台列表页堆叠。
- 每个页面都要有“下一步动作”。

### 1.2 低学习成本，少配置先可用
- 用户第一次打开就应能理解页面角色。
- 复杂配置应被收纳到设置页，不应侵入主流程。
- AI 能力默认有推荐策略，但允许高级用户深入配置。

### 1.3 解释优先于炫技
- 不只给计划结果，还要说明为什么这么安排。
- 不只显示画像字段，还要说明哪些字段正在影响计划。
- 对话产生的重要结构化变更必须可预览、可确认。

### 1.4 情绪负担要低
- 复盘页强调调整与恢复，不强调惩罚感。
- 首页提醒强调“可处理项”，而不是制造压力。
- 文案和交互都避免 PM / 管理后台式压迫感。

### 1.5 主路径始终围绕个人学习闭环
围绕以下链路组织产品：
1. 认识我是谁（画像）
2. 我要学什么（目标）
3. 为什么这样安排（计划）
4. 今天先做什么（执行）
5. 做完后怎么调（复盘 / 对话）

## 2. 本地优先，但不排斥外部模型
- 用户的画像、目标、计划、对话记录默认以本地数据为主。
- 模型能力是增强层，不是产品本体。
- 即使模型不可用，客户端仍应保留可浏览、可编辑、可理解的本地状态。

## 3. 模型策略是“可替换能力层”，不是页面中心
- 主界面不暴露过多模型细节。
- Provider、模型名、路由策略、API Key 等集中放在设置页管理。
- 业务页面只关心“当前能力是否可用、由谁提供、用于什么场景”。

## 4. 当前实现边界
当前阶段不追求：
- 完整后端服务化
- 复杂多代理编排
- 云端同步
- 课程内容平台化

当前阶段优先推进：
- 可接手的产品文档
- 稳定的数据结构与状态层
- 真实页面骨架
- 多模型 Provider 的抽象与设置入口
