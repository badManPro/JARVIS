# PRD · Learning Companion（学习伴侣）

## 1. 产品定位

Learning Companion 是一个 **本地优先的 AI 学习规划桌面应用**。

它不是课程平台，不是普通待办工具，也不是只会聊天的 AI 助手。它的核心能力是：
- 先通过画像和行为倾向理解用户
- 再基于目标、时间和状态生成个性化学习路径
- 在用户真正开始学习时，产出足够细的执行方案
- 在执行过程中依据任务状态和复盘结果持续调整后续计划

产品当前要解决的不是“记录任务”，而是“替用户减少学习决策成本”。

## 2. 核心价值

1. **理解用户**
   - 通过年龄段、MBTI、学习偏好、压力反应和反馈偏好建立动态画像。
2. **替用户拆解计划**
   - 先生成粗版学习路径，再在当天生成超细执行单。
3. **减少选择成本**
   - 尽量替用户决定今天学什么、先学什么、去哪里学、做到什么算完成。
4. **持续调整**
   - 基于完成、延期、跳过和复盘信号重排后续计划。
5. **本地优先**
   - 强调隐私、长期陪伴感和桌面端沉浸体验。

## 3. 目标用户

### 3.1 泛自学用户
- 想学新技能，但不会自己拆路径
- 容易卡在“今天该学什么”
- 需要低摩擦启动和持续陪伴

### 3.2 兴趣驱动型用户
- 可能同时学多个方向
- 容易被新兴趣吸引，主线不稳定
- 需要系统帮助保持主目标连续推进

### 3.3 需要个性化节奏的人
- 不同人格、状态和压力反应会明显影响计划执行
- 不希望所有计划都被模板化处理

## 4. 首发领域策略

产品长期是通用学习规划工具，但首发不追求“所有领域都同样深”。

首发深做领域：
- 编程
- 乐器
- 健身

原因：
- 三者都适合拆成阶段路径与日执行单
- 三者都能明确资源、练习和完成标准
- 三者都能体现画像与时间预算对计划的影响

后续扩展更多领域时，优先走：
- 通用规划内核不变
- 通过领域规则、资源规则和 prompt few-shot 扩展新领域

## 5. 核心问题

- 用户知道自己“想学什么”，但不会拆成可执行计划
- 用户不知道真实适合自己的节奏和反馈方式
- 计划常常脱离人的状态，只考虑目标不考虑执行摩擦
- 用户管理多个目标时容易分散，主线持续性差
- 细计划往往不够细，仍然要求用户自己找资料、自己决定步骤

## 6. 产品目标

### 6.1 建立动态用户画像
通过首次建档、持续问答、任务状态和复盘反馈建立长期画像。

### 6.2 把计划分成粗版与细版两层
- 粗版负责阶段目标、周里程碑和关键节点
- 细版负责当天真正要执行的步骤、资源、练习和交付物

### 6.3 支持一个主目标和多个副目标
系统默认保证主目标连续推进，副目标占用剩余时间。

### 6.4 让 AI 具备有限自治权
- 小调整自动执行
- 大调整先确认

### 6.5 让今日页成为真正的执行页
不仅展示今天的计划，还支持步骤级状态流转和后续重排。

## 7. 核心产品概念

### 7.1 动态画像
画像不是一次性表单，而是一个长期更新的规划基础。

采集与维护的重点字段：
- 年龄阶段
- MBTI
- 时间预算
- 最佳学习窗口
- 节奏偏好
- 动机方式
- 压力反应
- 反馈偏好
- 性格关键词

在首轮建档后，系统还要形成一层“规划倾向确认”，明确：
- 系统会如何拆任务
- 系统会如何给反馈
- 系统何时自动替用户做决定

### 7.2 粗版学习路径
粗版路径只回答长期问题：
- 这轮学习要经历哪些阶段
- 每周的里程碑是什么
- 当前最重要的关键节点是什么

### 7.3 细版执行单
细版计划只在当天或接近执行时生成，尽量细到不需要用户二次思考。

例如：
- 编程：今天学哪些语法、做几个例子、看哪份官方文档、写出什么最小脚本
- 乐器：今天练哪几段、多少 BPM、重复几次、是否录音回听
- 健身：今天练哪些动作、几组几次、组间休息多久、训练后如何恢复

### 7.4 主副目标调度
系统默认采用：
- 一个主目标
- 多个副目标

调度原则：
- 主目标优先占用稳定时间块
- 副目标填补剩余时间
- 时间变少时优先保主目标
- 尽量减少同一天内的频繁切换

### 7.5 有限自治
系统默认策略：
- 小调整自动执行
- 大调整征求确认

自动执行的小调整示例：
- 今天时间减少，自动压缩细版计划
- 某一步延期，默认顺延到明天候选区
- 状态差时自动降难度并保留主线

需要确认的大调整示例：
- 切换主目标
- 推翻当前粗版路径
- 重新分配一周主副目标比重

## 8. 关键模块

### 8.1 用户画像模块
负责：
- 首次建档
- 画像维护
- 规划倾向确认
- 解释系统如何使用这些信息

### 8.2 学习目标模块
负责：
- 创建多个学习目标
- 标记主目标和副目标
- 填写基础、周期和成功标准
- 维护目标状态

### 8.3 规划引擎
输入：
- 用户画像
- 主副目标
- 时间预算
- 学习窗口
- 历史执行信号
- 复盘结果

输出：
- 粗版阶段路径
- 周里程碑
- 今日细版执行单
- 调整建议
- 日历排程结果

### 8.4 今日执行引擎
今日页不只是展示计划，而是管理执行过程。

每个步骤支持状态：
- 待开始
- 进行中
- 完成
- 延期
- 跳过

已确认规则：
- 延期默认顺延到明天候选区
- 完成一步后自动把下一步提升为主焦点
- 跳过一步时，如果影响后续依赖，需要提示并压缩或重排后续步骤

### 8.5 日历与调度模块
负责：
- 让用户看到主副目标如何被安排到一周时间块
- 展示延期任务如何进入后续排程
- 让用户理解系统为什么这么安排

### 8.6 对话式调整模块
用户可通过自然语言表达变化，例如：
- 我最近很累，计划调轻一点
- 我这周只能练 20 分钟
- 我想把健身降级成副目标
- 我今天只想完成最低阻力版本

模块边界：
- 教练抽屉负责收集变化和产出建议
- 粗版路径的确认与重生成在 `学习路径`
- 今日细版计划的确认与重生成在 `今日`

## 9. 页面结构

建议一级导航：
1. 今日
2. 学习路径
3. 学习档案
4. 日历
5. 设置

### 9.1 今日
- 输入今天可用时间与限制
- 生成细版执行单
- 执行步骤状态流转
- 展示资源、练习和今日交付物

### 9.2 学习路径
- 查看粗版路径
- 查看阶段与里程碑
- 管理主副目标关系
- 触发粗版计划重生成

### 9.3 学习档案
- 查看和编辑画像
- 查看系统当前对用户的理解
- 查看画像如何影响规划和反馈方式

### 9.4 日历
- 查看一周排程
- 看主副目标如何分配时间
- 看延期任务如何被顺延

### 9.5 设置
- Provider、模型和路由策略
- 自动调整偏好
- 资源偏好与提醒风格

## 10. MVP 范围

### MVP-1 首次建档升级
- 补齐画像字段
- 生成规划倾向确认
- 明确解释系统如何基于画像工作

### MVP-2 今日页执行化
- 细版计划改成步骤级结构
- 支持开始 / 完成 / 延期 / 跳过
- 延期默认进入明天候选区

### MVP-3 主副目标与日历调度
- 支持一个主目标和多个副目标
- 支持基础日历排程视图
- 支持自动保主线

### MVP-4 首发领域深做
- 编程规则
- 乐器规则
- 健身规则
- 对应资源推荐策略

### MVP-5 强反馈交互
- 计划生成动画
- 状态流转反馈
- 自动推进下一步的焦点提示

### MVP-6 桌面角色反馈层
- 提醒
- 庆祝
- 陪伴反馈

> 注：桌面动漫形象属于强化体验方向，不是当前主流程 MVP 的阻塞项。

## 11. 成功标准

- 用户完成首次建档后，能感知系统“理解了我”
- 用户可以管理多个目标，但主目标始终清晰
- 系统能输出非模板化的粗版路径
- 今日页能生成真正可执行的细版计划，而不是泛泛建议
- 用户完成、延期、跳过步骤后，系统能进入后续调度逻辑
- 编程、乐器、健身三类领域的计划明显不同，不像同一套模板

## 12. 产品原则

1. 先理解用户，再给计划
2. 计划必须足够细，尽量替用户做选择
3. 粗版路径和今日细版计划必须分层
4. 多目标存在，但只能有一个主目标
5. 小调整自动执行，大调整再确认
6. 今日页必须是执行页，而不是只读展示页
7. 界面优先回答“我现在该做什么”
8. 产品优先做“懂人 + 落地”，而不是内容平台
9. 模型能力是可替换能力层，不把产品绑死在单一厂商

## 13. 模型策略

### 13.1 多 Provider 支持
产品继续保持多模型 Provider 抽象，首批明确考虑：
- GPT / OpenAI
- GLM / 智谱
- Kimi / Moonshot
- DeepSeek

### 13.2 用途路由，而非厂商直连
业务模块按用途描述能力：
- 画像提取
- 粗版计划生成
- 今日细版计划生成
- 计划调整
- 复盘总结
- 通用对话

再由设置中的模型策略把这些用途映射到对应 Provider。

### 13.3 领域扩展策略
后续支持更多学习领域时，优先通过以下方式扩展：
- 领域规则
- prompt 模板
- few-shot 示例
- 推荐资源来源

不推翻通用规划内核。

## 14. 当前实现边界

当前代码基线已经具备：
- Electron + React + TypeScript 桌面壳
- 本地 SQLite 持久化
- 学习档案、学习路径、今日页、设置页骨架
- 基础 goal / plan / conversation / provider 持久化
- AI capability 抽象与基础运行时

当前尚未完成的关键产品能力：
- 画像驱动的规划倾向确认
- 真正步骤级的今日执行模型
- 主副目标调度与日历视图
- 三个首发领域的深度规则
- 桌面角色反馈层

## 15. 暂不做

- 云同步
- 多人协作
- 通用内容平台化运营
- 所有学习领域的全面覆盖
- 把桌面角色作为第一阶段主入口
